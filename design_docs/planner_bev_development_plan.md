# Planner 改造・BEV 表現 開発方針メモ

本ドキュメントは、現状認識・今後の方針・および機械学習ベースの planner を見据えた「BEV（画像ではない）」表現の一般的な整理をまとめたものです。必要に応じて追記・分割してください。

---

## 1. 現状

- **走行**: 問題なく走行できているが、**回避や停止の挙動は未実装**（常に追従・通過寄りの挙動になっている）。
- **Localization**: 実用上ほぼ問題ないレベルに感じられる。
- **制御（MPC）**: 与えられた経路に対しては **ほぼ問題なく追従**できている印象。
- **検知**: 他車の座標がトピックとして届くため、当面は **その情報を参照する**方針でよい。
- **トラッキング**: **未実装**（同一物体の時系列 ID 付き追跡などはこれから）。

---

## 2. 今後の方針（高レベル）

- **Planner を大幅に改造**する。回避・停止・将来軌道の安全性などは planner 側の責務として明確化したい。
- その土台として、環境を **BEV 形式の perception / scene 表現**に落とし込む。
- **経路・マップ・検知**は既にシステム側で揃っている想定のため、**ルールベースでも学習ベースでも同じテンソル／構造体に射影できる**ことを最初の設計目標にする。

---

## 3. 機械学習 planner を見据えた「BEV = 画像ではない」とは何か

カメラ RAW をそのまま BEV に射影した「見た目の画像」とは区別し、**平面（または薄い 3D）上の離散格子＋チャネル**、あるいは **ベクトル／クエリ**として場を表すのが一般的です。以下は文献・実務でよく見られるパターンです。

### 3.1 ラスター（マルチチャネルテンソル）型 — 最も一般的な「非画像」BEV

- **形状のイメージ**: `C × H × W`（または `H × W × C`）。**RGB ではなく意味チャネルの積み重ね**。
- **各セル**は、車体座標またはマップ座標で一定サイズ（例: 0.1 m / cell）の地面タイルに対応。
- **チャネルの例**（論文・オープンソースで多い組み合わせ）:
  - **占有（occupancy）**: 自由空間 / 占有の二値または確率。
  - **セマンティック**: 道路、歩道、レーン、静止障害物などの one-hot または softmax 前ログit。
  - **道路構造**: 中心線からの横ズレ、進行方向、曲率に相当するスカラー場（SDF / 距離場 / lane direction field など）。
  - **動的物体**: 他車・歩行者の占有確率、**速度場**（セルごとの vx, vy）や将来時刻のスタック（複数フレームをチャネル方向に積む）。
  - **ego 関連**: 自車位置・向きを原点に合わせたマスク、計画ホライズン内の参照経路をチャネルとして焼き込む、など。
- **学習 planner との接続**: CNN / U-Net / 小型 ResNet、BEVFormer 系の **BEV エンコーダ**、Transformer で flatten して sequence 化、など。**「画像」として可視化はできるが、中身はセマンティック・幾何のスタック**と捉えるのが実務的。

参考（占有格子と計画の古典的つながり）: [Driving among Flatmobiles (arXiv)](https://arxiv.org/abs/2008.04047) — BEV occupancy と軌道計画の議論。

### 3.2 高さ方向を持つ BEV / 薄い 3D ボクセル

- 2D BEV に **複数の高さビン**を持たせる（擬似 3D occupancy）。立体駐車場や橋の上下関係がないレース環境では省略しやすいが、**一般化**するなら選択肢に入る。

参考: [HBEVOcc などの height-aware BEV](https://pmc.ncbi.nlm.nih.gov/articles/PMC12899489/)（カメラ起点だが、出力表現としての「H×W×Z ビン」は planner 入力設計の参考になる）。

### 3.3 ベクトル化（インスタンス列）— 計算効率・解釈性のトレードオフ

- エージェント・レーン・停止線などを **ポリライン・バウンディングボックスのパラメータ列**として表現（Transformer の object query に近い）。
- **BEV グリッドを持たない**ためメモリ効率が良く、最近の motion forecasting / planning で増えている。

参考: [VAD: Vectorized Scene Representation for Efficient Autonomous Driving](https://arxiv.org/abs/2303.12077)

### 3.4 クエリベース占有・スパース表現

- 全域の dense グリッドを予測せず、**プランナが必要とする時空間点だけ occupancy を問い合わせる**設計。解釈性と計算量のバランスを取る方向。

参考: [QuAD: Query-based Interpretable Neural Motion Planning](https://arxiv.org/abs/2404.01486)

### 3.5 離散トークン・軌道指向 BEV

- BEV を **離散トークン列**にして自己回帰 planner に渡す、あるいは **軌道ヒートマップ**など計画タスクに直結した BEV ターゲットを使う、など多様。

参考: [DAP (Discrete-token Autoregressive Planner)](https://arxiv.org/html/2511.13306v1)、[TrajDiff 等の議論](https://arxiv.org/abs/2512.00723)

### 3.6 設計上の注意（planner 特化の観点）

- 高解像度の「知覚用 BEV」をそのまま planner に入れると、**クローズドループ性能や過学習**の観点で最適とは限らない、という整理もある。**計画に必要なチャネル・解像度・ホライズン**に絞るのが実務的。

参考: [What Matters for Scalable and Robust Learning in End-to-End Driving Planners?](https://arxiv.org/html/2603.15185v1)

## 4. 本プロジェクトへの当てはめ（提案）

| 観点 | 提案 |
|------|------|
| 短期（ルールベース planner 強化） | 既存トピック（他車位置、経路、マップ）から **`C×H×W` のマルチチャネル BEV テンソル**（または同等の構造体）を生成するノードを一つ挟み、**回避・停止ロジックをこの表現上で記述**する。 |
| 中期（学習 planner） | 同じテンソル（または同じ情報を並べたベクトル化表現）を **教師データ付きで保存**し、軌道または速度プロファイルを出力するネットに接続する。 |
| トラッキング未実装 | BEV の「動的チャネル」は当面 **検知トピックの瞬時位置をラスタライズ**でも可。将来、トラッキングで **速度・予測軌道チャネル**を足す。 |
| シミュレータ特性 | レースカート・平坦路面なら **2D BEV + 占有・レーン・参照経路・他車占有**で十分なことが多い。 |
