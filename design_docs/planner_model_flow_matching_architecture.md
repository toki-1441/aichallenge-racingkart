# Planner モデル方針: BEV → 多層特徴 → Flow Matching 多軌道 → 選択 → MPC

本稿は、既存の **BEV シーン表現（`bev_scene_stack` 等）** を前提に、次段の **学習型 planner モデル**の構成案を整理したものです。ユーザー案（高品質 encode → 中間表現で多層特徴抽出 → flow matching で複数軌道 → 選択 → MPC）を満たしつつ、文献・近傍領域から得られる **よりよい設計の選択肢** を併記します。

**更新方針（合意事項）**

- **BEV の位置づけ**: 他モダリティ（V2X・参照経路など）を併用してよいが、本プロジェクトの BEV は **幾何・レーン・他者占有を明示的チャネルにしたラスタ**であり、**正確性（信頼できる空間整合）が主な利点**である。学習モデルは「曖昧な画像 BEV」より **この高忠度テンソルを主信号**として扱う前提で進める。
- **開発の着手フェーズ**: **P1 から開始**する（下記 §5）。P0 は必須前段とはしない。
- **候補選択**: **ルールベースのスコア（解析コスト）で K→1 を決める**方針とする。学習型スコアラーは当面オプションとする。

---

## 1. 要件の再整理（ユーザー案）

| ブロック | 役割 |
|----------|------|
| **Encoder** | BEV（および将来の追加特徴）を、**精度が出やすい構造**で圧縮・正則化する |
| **中間表現 + 多層** | 幾何・相互作用・長期文脈などを **段階的に** 取り出す |
| **Flow Matching Decoder** | **複数本**の将来軌道（または制御系列）を生成 |
| **選択** | **ルールスコア**（占有干渉・レーン逸脱・滑らかさ等の加权和）で **1 本（または少数）** に絞る |
| **MPC** | 選ばれた軌道を **参照・初期値・コスト項** として取り込み、硬い制約下で追従 |

このパイプラインは **「生成（ソフト）」と「制約充足（ハード）」の分離**が明確で、レーシングカートのような **低レイテンシ MPC** とも相性が良いです。

## 2. 関連研究・トレンド（要点とリンク）

### 2.1 Flow Matching を軌道生成に使う系

- **Trajectory Conditional Flow Matching (T-CFM)** — 軌道予測・生成を flow matching で統一し、拡散系と比較して **推論コストを大きく削減**しつつ精度を維持する方向の報告があります。  
  - [arXiv:2403.10809 — Efficient Trajectory Forecasting and Generation with Conditional Flow Matching](https://arxiv.org/abs/2403.10809)

- **GoalFlow (CVPR 2025)** — ゴール駆動で **マルチモーダル軌道**を flow matching で生成。目標点のスコアリングと組み合わせ、E2E 系ベンチで強い結果。  
  - [GoalFlow (arXiv)](https://arxiv.org/abs/2503.05689)

- **GuideFlow** — 安全性・物理制約を **生成過程に直接組み込む** constraint-guided flow matching。後処理だけに頼らない設計は、**MPC 前段の候補品質**を上げる観点で参考になります。  
  - [GuideFlow (arXiv)](https://arxiv.org/abs/2511.18729)

**示唆**: 素朴な「decoder が K 本吐く → 後で全部 MPC で弾く」より、**学習時・推論時に制約寄りの誘導**（GuideFlow 型、または下記 4.3 のマスク/clip）を入れると、**ルールスコアで弾く違反が減り**、選択と MPC が安定しやすいです。

### 2.2 BEV と予測・計画の接続

- **Direct BEV Feature Attention** — オンラインマップ推定の **内部 BEV 特徴**を予測ヘッドに直接つなぎ、推論と精度の両立を図る例。  
  - [arXiv:2407.06683](https://arxiv.org/abs/2407.06683)

**示唆**: 現在の **離散 BEV テンソル**に加え、将来的に **CNN/Transformer の中間特徴マップ**（解像度を落としたマルチスケール）を明示的に保持すると、encoder の「精度が出やすい構造」になりやすいです。

### 2.3 ニューラルと MPC の接続

- **PlanNetX** — MPC が生成する最適軌道構造を損失に取り入れ、**NN が MPC 的な軌道を学習**する枠組み。  
  - [PlanNetX (PMLR)](https://proceedings.mlr.press/v242/hoffmann24a.html)

- **Planning with Adaptive World Models** 等 — 予測世界モデル + MPC の階層（nuPlan 系）。  
  - [arXiv:2406.10714](https://arxiv.org/abs/2406.10714)

**示唆**: 最終段を MPC に固定するなら、**教師信号を「人間デモ」だけでなく「MPC + 現在のコストモデル」から取る**、あるいは **残差学習**（NN は MPC 解への補正のみ）にすると、閉ループで破綻しにくいです。

## 3. アーキテクチャ案（ユーザー案を満たす具体化）

### 3.1 Encoder（BEV → 高信頼表現）

**目的**: 下流が「幾何・可通行性・他者」を誤解しない表現を得る。

**本リポジトリの BEV の利点（設計上の主張）**

- **空間的に正確**: Lanelet OSM 由来のレーン、自車・V2X 由来の占有などを **同一 ego 格子・同一解像度**に載せているため、学習データと推論時の **幾何アラインメントが崩れにくい**。
- **解釈可能**: チャネル意味が固定（レーン / 経路 / 障害 / ego 等）であり、損失設計・ルールスコアの **ターゲットチャネルが明示的**。
- **下流の負荷低減**: 「場の理解」を BEV で済ませ、decoder は **軌道の多様化**に集中できる（幻覚したレーンをネットが捏造しにくい）。

他モダリティは **補助**（速度・ヨーレート、抽象ゴール、将来のトラック履歴など）として concat してよいが、**主幹は高忠度 BEV** とする。

| 選択肢 | 内容 |
|--------|------|
| **ConvNeXt / U-Net 系** | `C×H×W` に強く、実装・デバッグ容易。マルチスケール skip が自然に「多層中間表現」に接続できる。 |
| **Swin / ViT-BEV** | 大域依存を取りやすい。データが増えた段階で有力。 |
| **ハイブリッド** | 浅い Conv でノイズ除去 → Transformer で長距離依存（レーン分岐・対向車線）。 |

**精度を上げる実務テクニック**（encoder 周り）:

- **幾何整合**: 自車原点・解像度を学習時と推論時で固定し、**データ拡張は回転・シフトを物理的に妥当な範囲**に限定。
- **クラス不均衡**: 占有・レーンは希少チャネルになりがち → **focal loss / balanced sampling**。
- **マルチ解像度入力**: 粗いグローバル BEV + 細い ego 近傍 crop の **2 ストリーム**（計算は増えるが衝突回避に効きやすい）。

### 3.2 中間表現・多層特徴抽出

**目的**: 単一の flatten ベクトルにしないことで、decoder が **「どこを見るべきか」** を段階的に利用する。

| パターン | 説明 |
|----------|------|
| **FPN 型** | 解像度の異なる特徴マップを保持し、decoder で **ピラミッド融合**（検出・セグ系の定石）。 |
| **Latent tokens** | BEV を `N` 個のクエリに集約（DETR 系 / Perceiver）。軌道 head は **軌道クエリ K 本**とクロスアテンション。 |
| **明示的な「レーン・エージェント」クエリ** | レーングラフや V2X 他車を別モダリティとして **融合ブロック**に投入（後述 4.1）。 |

ユーザー案の「いくつかの層で特徴抽出」は、実装上は **バックボーンの stage2 / stage3 / stage4 を別々にデコーダへ渡す**か、**Transformer ブロックを 2〜3 スタックに分けて中間をスキップ接続**する形で満たせます。

### 3.3 Flow Matching Decoder（K 本軌道）

**出力の表現**（どれを生成するか）:

| 表現 | メリット | デメリット |
|------|----------|------------|
| **平面 (x,y) 軌道** | MPC の参照経路と直結しやすい | 曲率・速度の不整合を別処理で補正 |
| **(x,y,θ,v) 状態列** | 運動学と整合しやすい | 次元増、学習難 |
| **制御系列 (δ, a)** | 低レベルに近い | 積分誤差・MPC 再最適化との二重化 |

**推奨（本プロジェクト向け）**: まず **地平面上の (x,y) リサンプリング済み軌道**（ホライズン `T`、ステップ `Δt`）を flow の出力とし、**MPC 側で速度プロファイルと曲率制約を満たすようフィット**する。既存 MPC が強いなら **NN は幾何的「行きたい通路」**に専念させるのが安全です。

**Flow の条件付け**: encoder 出力 + **自車速度・ヨーレート** + **目標（次コーナー exit / ラップタイム用の抽象ゴール）** を concat または AdaLN で注入。

### 3.4 選択（K → 1）— **ルールスコアを採用**

**方針**: 候補軌道ごとに **同じルールで数値スコア**を計算し、**最小化（または最大化）で 1 本を選ぶ**。学習は軌道生成側に寄せ、選択は **検証可能・調整可能**な解析コストに寄せる。

| ルール項（例） | 概要 |
|----------------|------|
| **占有・障害干渉** | BEV の障害チャネル（および必要なら同一テンソル上の他チャネル）に対し、軌道サンプル点から **最近傍距離ペナルティ**や **侵入二値ペナルティ**を積分。 |
| **レーン・可走域** | レーン（または自由空間）チャネル外への逸脱に **距離ペナルティ**。 |
| **幾何的滑らかさ** | 曲率・曲率変化（数値差分）の L2 / ハード上限違反ペナルティ。 |
| **レース目的（任意）** | ラップタイム向けなら **進行方向の前進報酬**、保守走行なら **中央寄与**などを重み付きで加算。 |

**実装メモ**

- スコアは `S = Σ w_i c_i` とし、`w_i` を YAML でチューニング可能にすると運用しやすい。
- **同点・タイブレーク**は占有違反 > レーン逸脱 > 滑らかさ の **辞書式**でもよい。

**将来オプション（当面やらない）**

- **学習型スコアラー**（値関数・ranking）は、ルールスコアで足りなくなった段階で検討。
- **各候補にフル MPC を回して順位**する方式は遅延が大きいため、必要になれば **短いホライズン・粗い離散化**に限定して検討。

**文献との関係**: GoalFlow の **ゴール点の離散化 → 条件付け生成**は、**K 本の「意味的モード」を減らす**参考になる。ただし **勝者決定は本プロジェクトではルールスコアに統一**し、ゴールスコア学習に依存しない。

### 3.5 MPC への渡し方

| モード | 説明 |
|--------|------|
| **参照経路** | 選んだ軌道を `waypoints` または `reference_line` として既存インターフェースに投入。 |
| **ウォームスタート** | MPC の **初期制御系列**を NN 出力からコピー（次元が合う場合のみ）。 |
| **コストシェイピング** | レーン中央・障害物距離の **重み場**を BEV から作り、MPC のステージコストに加算（VLM-MPC 系の発想の軽量版）。 |

## 4. 「よりよい」発展アイデア（要件を維持したまま強化）

### 4.1 マルチモーダリティ（BEV 以外の入力）

**BEV 以外の入力も問題ない**が、**主信号は高忠度 BEV**（§3.1）。補助ブランチで次を足すと、相互作用や長期文脈の取りこぼしを減らせる。

- **V2X / 他車 Pose**（既存トピック）: 小型 MLP + 相対幾何で **エージェントトークン**化（BEV 占有と冗長でも相補的になり得る）。
- **レーン参照線**（従来 planner の出力）: **曲率・横偏差**をスカラー系列として concat。
- **将来**: トラッキング ID 付き履歴 → **Temporal Transformer**。

### 4.2 拡散ではなく Flow を選ぶ理由の整理（設計判断メモ）

- **サンプリング回数が少なくても品質が出やすい**という報告が flow matching 側に多い（T-CFM 等）。
- レース用途では **50 Hz 近い再計画**が欲しくなるため、**ステップ数を抑えた rectified flow / 1〜数ステップ推論**の検証価値が高いです。

### 4.3 制約を「選ぶ前」に入れる（GuideFlow 精神の軽量版）

フル GuideFlow までは行かずとも、

- デコード中に **占有マスクで logits をマスク**（不可走領域への流れを抑止）、
- または **後処理で clip したうえで ranking**、

のどちらかで **MPC 前の破綻率**を下げられます。

### 4.4 不確実性の明示（レース向けオプション）

- **同一条件でノイズのみ変えた複数サンプル** → 分散が大きいシーンでは **ルールスコア側で保守項の重みを上げる**など、選択ルールで表現してもよい。
- あるいは **flow の途中特徴から予測分散**を別 head で出し、**ルールの重みを状態依存**にする（学習は分散 head のみ、選択は依然ルール系）。

### 4.5 学習データの閉ループ

- **シミュログ + MPC ラベル**（PlanNetX 系）で教師を増やす。
- **オンライン悪例のみファインチューン**（DAgger 系）でドリフト抑制。

## 5. 段階的ロードマップ（**P1 から着手**）

**前提**: 単一軌道の最小パイプライン（旧 P0 に相当）が既に不要、または別途満た済みとし、**マルチ候補 + ルール選択から直接始める**。

| フェーズ | 内容 | 成果物イメージ |
|----------|------|----------------|
| **P1（現在の着手点）** | **Encoder（小型 Conv / FPN）** + 中間層で特徴抽出 + **K 本並列ヘッド**（教師は k-means / 多峰 BC / MPC ログから得た **K モード**）+ **ルールスコアで 1 本選択** + MPC 接続 | マルチ軌道パイプライン・ルール重みのチューニング基盤 |
| **P2** | **Conditional Flow Matching**（または rectified flow）に **K 本生成ヘッドを置換**；選択は **引き続きルールスコアのみ** | 生成品質の向上・サンプル多様性 |
| **P3** | 制約誘導（マスク / clip）+（任意）MPC からの蒸留・データ拡張 | 閉ループ性能・安全性の仕上げ |

**P0（任意・省略可）**: 単一 MLP 軌道 + MPC のみのスモークテスト。初めて学習パイプラインを繋ぐ場合のみ短期で実施。

**P1 で押さえる検証項目（例）**

- BEV 解像度・チャネル固定下で、**ルールスコアと人間判断（ログ比較）の相関**が取れるか。
- K を増やしたときの **レイテンシ**（スコアは O(K・T・軽い幾何) を維持）。
- MPC が受け取る参照の **インターフェース契約**（メッセージ型・再計画周期）。

## 6. 既存リポジトリとの接続メモ

**入力・損失の確定仕様**は [`planner_training_inputs_and_losses.md`](./planner_training_inputs_and_losses.md)（BEV 形状 `4×256×144`、正規化、P1 の `L_pose` / オプション `L_div` / `L_κ`、P2 の FM 予告）に集約する。  
**実装コード**は [`aichallenge/ml_workspace/planner_bev/`](../aichallenge/ml_workspace/planner_bev/)（`train.py`, `lib/model.py`, `lib/loss.py`, `prepare_data.py`, `lib/rule_score.py`）。

- **入力**: `bev_scene_stack/tensor`（`Float32MultiArray`）は既に **ego 座標系ラスタ**なので、encoder の前処理（reshape、`C×H×W`）がそのまま定義できる。**ルールスコアも同一テンソル（または同期したコピー）を参照**すれば、学習時と車載で一貫した幾何判定が可能。
- **他車**: 同一チャネルに V2X を焼き込み済みなら **そのまま入力**も可能だが、**生 Pose を別入力**にすると相互作用が学習しやすい（4.1）。
- **出力**: 既存 `multi_purpose_mpc_ros` 等が受け取れる **waypoint / path メッセージ**に合わせた **アダプタノード**を別途用意すると、学習コードと車両ソフトの境界が明確になる。

## 7. まとめ

- **BEV は高忠度・解釈可能な主信号**とし、その正確性を encoder と **ルールスコア**の両方で活かす。
- 開発は **P1（多層特徴 + K 本ヘッド + ルール選択 + MPC）から開始**し、安定後に **P2 で Flow Matching に置換**する二段ロケットがリスク分散しやすい。
- ユーザー案の **encode → 多層中間 → flow 多軌道 → 選択 → MPC** は、近年の **flow 系軌道生成（T-CFM / GoalFlow / GuideFlow）** と整合的。**選択は学習スコアラーではなくルールスコアで統一**する。
- **よりよくする**ための補助打ち手は、(1) **BEV を主としたマルチ入力**（4.1）、(2) **選択前の clip / マスク**（4.3）、(3) **MPC / シミュからの教師**（4.5）、(4) **ゴール条件付けで K の意味を構造化**（3.4 脚注）、が扱いやすい。

本稿は `design_docs/planner_bev_development_plan.md` の「モデル・学習」側の補助ドキュメントとして位置づけ、実験結果に応じて更新してください。
